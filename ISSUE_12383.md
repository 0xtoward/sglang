# Issue #12383: VLM Token 不一致性问题

## 问题核心

**当使用 `input_ids` + `image_data` 输入时，SGLang 会执行不必要的 detokenize-retokenize 循环，导致原始 token 序列被改变，破坏 RLHF 训练所需的 token 一致性。**

## 问题表现

- **输入**: 50 tokens，包含 2 个 `<|image_pad|>` token (151655)
- **输出**: 290 tokens，包含 242 个 151655 tokens
- **结果**: Token 序列长度和内容完全改变，Hash 值不同

## 根本原因

### 1. 代码路径

```
tokenizer_manager.py: _tokenize_one_request()
  ↓ 传入 input_ids (List[int])
base_processor.py: load_mm_data()
  ↓ 检测到 List[int]，强制 decode 为文本
base_processor.py: process_mm_data()
  ↓ 调用 HuggingFace processor.__call__() 重新 tokenize
  ↓ 根据图像分辨率动态扩展 <|image_pad|>
```

### 2. 关键问题点

- **`base_processor.py:476`**: 无条件将 `List[int]` decode 为文本
- **`base_processor.py:275`**: 调用 HuggingFace processor 重新 tokenize
- **动态扩展**: HuggingFace processor 根据图像分辨率（如 300x300）将单个 `<|image_pad|>` 字符串扩展为多个 151655 tokens

### 3. 为什么是问题

RLHF 训练需要：
- 精确的 token 序列：每个 token 位置必须一致
- Logprobs 对齐：`logprob[i]` 必须对应 `input_ids[i]`
- 奖励计算：基于 token-level logprobs

当前问题导致：
- Token 位置偏移（如 token_10 从索引 10 变成 130）
- Logprobs 错位（logprob[10] 对应错误的 token）
- 奖励计算错误

### 3. 日志观察


### 4. 数学验证

- 输入：2 个 `<|image_pad|>`，每个扩展为 ~120 tokens
- 扩展量：2 × 120 = 240 tokens
- 新长度：50 + 240 = 290 tokens ✓
- 新 151655 数量：2 + 240 = 242 个 ✓

## 解决方案方向

需要添加 "fast path"，避免不必要的 detokenize-retokenize：

1. **`load_mm_data()`**: 检测到 `List[int]` 输入时，不要 decode，直接保留 `input_ids`
2. **`process_and_combine_mm_data()`**: 如果 `base_output.input_ids` 存在，直接使用
3. **只提取图像特征**: 使用 `image_processor` 单独处理图像，不重新 tokenize 文本部分
4. **保持 token 一致性**: 确保输出的 `input_ids` 与输入的 `input_ids` 完全一致（除了图像特征部分）

## 对比：纯 LLM vs VLM

- **纯 LLM**: 检测到 `input_ids` 时直接使用，不走 detokenize-retokenize
- **VLM**: 即使输入是 `input_ids`，也强制 decode 并重新 tokenize，绕过了 fast path
